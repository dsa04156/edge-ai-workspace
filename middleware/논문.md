혼합 디바이스 엣지 AI 런타임 오케스트레이션 시스템 구현

요약

본 문서는 정책 제안 중심 논문이 아니라, 현재 edge-orch 시스템에서 실제로 구현 및 배포된 구성요소를 기술하는 구현 논문 초안이다. 대상 시스템은 KubeEdge/Kubernetes 기반 mixed-device testbed(x86 서버 2대, Jetson 1대, Raspberry Pi 5 1대)에서 동작하며, workflow-aware orchestration을 위해 state_aggregator, placement_engine, workflow_executor를 핵심 서비스로 구성한다. 구현의 목적은 이기종 노드 상태와 워크플로우 이벤트를 결합해 stage 단위 실행 결정을 내리고, 실제 Kubernetes Job 실행까지 연결하는 end-to-end 제어 루프를 제공하는 것이다.

주요어: Edge Orchestration, KubeEdge, Mixed Device, Workflow Execution, Implementation

1. 서론

혼합 디바이스 엣지 환경에서는 노드별 아키텍처, 성능, 네트워크 특성이 상이하여 단순 정적 배치로는 안정적인 서비스 품질을 유지하기 어렵다. 특히 AI 워크플로우를 capture, preprocess, inference와 같은 stage로 분해해 실행할 때, 노드 상태 변화에 따른 재배치 및 오프로딩 판단이 필요하다.

본 연구의 범위는 새로운 이론 모델 제안이 아니라, 실제 운영 가능한 런타임 오케스트레이션 시스템 구현이다. 이를 위해 다음을 구현 대상으로 삼았다.

- 상태 수집 및 정규화 서비스
- stage 배치 의사결정 엔진
- Kubernetes Job 기반 워크플로우 실행기
- 워크플로우 이벤트 보고 경로

2. 시스템 구성

2.1 테스트베드 및 클러스터 상태

실제 클러스터 확인 결과, edge 노드는 KubeEdge 버전 문자열을 갖는 kubelet로 조인되어 있다.

- etri-dev0001-jetorn: v1.32.10-kubeedge-v1.23.0, role=agent,edge
- etri-dev0002-raspi5: v1.32.10-kubeedge-v1.23.0, role=agent,edge
- etri-ser0001-cg0msb: control-plane
- etri-ser0002-cgnmsb: worker

또한 kubeedge 네임스페이스에서 cloudcore, edgemesh-agent, cloud-iptables-manager가 Running 상태로 확인된다.

2.2 상위 아키텍처

구현된 상위 흐름은 다음 루프를 따른다.

monitor -> normalize -> decide -> execute -> report -> summarize

입력은 Prometheus/node-exporter 메트릭과 workflow event이며, 출력은 stage 배치 결정 및 Kubernetes Job 실행 결과다.

3. 구현 상세

3.1 state_aggregator 구현

state_aggregator는 FastAPI 서비스로 구현되었으며 다음 API를 제공한다.

- POST /workflow-event
- GET /state/nodes
- GET /state/node/{hostname}
- GET /state/workflows
- GET /state/workflow/{workflow_id}
- GET /state/summary
- GET /metrics

내부적으로 Prometheus 메트릭 폴링과 Kubernetes 노드 동적 매핑을 수행하며, 노드 상태를 compute/memory/network pressure 및 node_health로 정규화한다. 워크플로우 이벤트는 urgency, sla_risk, placement_stability로 집계된다.

3.2 placement_engine 구현

placement_engine은 FastAPI + Python heuristic engine으로 구현되었다. API는 다음과 같다.

- GET /state/nodes
- POST /placement/decide
- POST /placement/replan

의사결정 함수는 node profile, node state, stage metadata, current placement를 입력받아 weighted score를 계산한다. 결과는 target_node, action_type(keep/migrate/offload_to_cloud/reject), decision_reason, score_breakdown을 포함한다. 초기 구현은 RL/LLM 제어 없이 규칙 및 가중치 기반으로 구성했다.

3.3 workflow_executor 구현

workflow_executor는 stage/workflow 실행 API를 제공한다.

- POST /execute/stage
- POST /execute/workflow
- GET /workflow/{workflow_id}
- GET /workflows

실행기는 placement_engine의 결정 결과를 반영해 Kubernetes Job을 생성 및 감시한다. stage_start, migration_event, stage_end, failure_event를 workflow-event로 전송하며, 실패 시 워크플로우 상태를 failed로 업데이트한다.

3.4 workflow_reporter 구현

workflow_reporter는 helper 모듈 형태로 구현되었고, 다음 이벤트 전송 함수를 제공한다.

- report_stage_start
- report_stage_end
- report_migration
- report_failure
- report_workflow_end

해당 경로는 state_aggregator의 POST /workflow-event와 직접 연동된다.

4. 배포 및 운영 상태

클러스터 실측 결과(default 네임스페이스 기준), 다음 서비스가 실제 배포되어 Running 상태임을 확인했다.

- state-aggregator Deployment 1/1, Service 8000/TCP
- placement-engine Deployment 1/1, Service 8001/TCP
- workflow-executor Deployment 1/1, Service 8002/TCP

즉, 본 시스템은 문서 수준 설계가 아니라 배포 가능한 제어 평면으로 구현되어 있으며, mixed-device KubeEdge 환경과 연동된 상태다.

5. 구현 결과 및 현재 한계

5.1 구현 결과

- 상태 수집-정규화-의사결정-실행-이벤트 피드백의 end-to-end 루프 구현
- KubeEdge edge 노드 포함 클러스터에서 실제 컴포넌트 실행 확인
- stage 단위 action_type 기반 제어(keep/migrate/offload/reject) 지원

5.2 한계

- workflow_reporter는 라이브러리 중심으로, 독립 장기 실행 서비스화는 추가 작업 필요
- handoff artifact plane(MinIO) 및 metadata plane(Redis)의 완전 통합은 후속 단계
- 성능 평가는 현재 구현 검증 위주이며, 대규모 반복실험 자동화는 확장 필요

6. 결론

본 구현 논문은 edge-orch가 mixed-device 환경에서 동작하는 실제 런타임 오케스트레이션 시스템임을 보였다. 핵심은 state_aggregator, placement_engine, workflow_executor의 연동을 통해 stage 단위 실행 제어를 수행하는 점이며, KubeEdge 클러스터에서의 실배포 상태를 통해 시스템 구현의 현실성을 확인했다. 향후에는 handoff 데이터 평면 통합(MinIO/Redis), 워크플로우 다양화, 장기 안정성 평가를 통해 운영 수준의 완성도를 높일 예정이다.
