# AGENTS.md

## 1. 프로젝트 개요

이 프로젝트는 KubeEdge 기반의 혼합 디바이스 엣지 AI 플랫폼을 대상으로 한다.

대상 환경:

* x86 엣지 AI 서버
* Jetson 엣지 AI 디바이스
* Raspberry Pi 5 엣지 AI 디바이스
* 물리 센서 디바이스
* 가상 디바이스 / 디바이스트윈

현재 단계의 1차 목표:

* 서비스 데모를 먼저 완성한다.
* 디바이스를 플랫폼에 연결한다.
* 디바이스와 서비스를 연결한다.
* 전체 운영 상태를 대시보드에서 한눈에 보이게 한다.
* 실제 현장 생산성 향상 효과를 설명 가능한 형태로 만든다.

장기 목표:

* 이후 동적 워크플로우 기반 플랫폼으로 확장한다.
* 런타임 오케스트레이션, 가상화, 스케줄링, 오프로딩 고도화를 단계적으로 반영한다.

중요한 점:
현재 단계에서 이 프로젝트는 “오케스트레이션 로직이 얼마나 복잡한가”로 먼저 평가받지 않는다.
지금은 아래를 먼저 보여줘야 한다.

* 디바이스가 실제로 등록되고 관리되는가
* 디바이스와 서비스가 실제로 연결되는가
* 전체 관계와 상태가 대시보드에서 보이는가
* 이 플랫폼이 현장에서 왜 유용한가

## 2. 현재 전략 해석

공식 과제 문서상 세부1의 핵심 범위는 여전히 아래와 같다.

* AI 서비스 분할 및 오프로딩
* AI 서비스 배포 및 통합 자원 관리 워크플로우
* 고신뢰 실시간 데이터 전송 미들웨어
* 온디바이스 가상화
* 멀티 디바이스 오케스트레이션
* 디바이스 성능 강화

하지만 현재 내부 우선순위는 아래처럼 재정렬한다.

1. 서비스 데모 우선
2. 디바이스 연결 구조 정의
3. 디바이스-서비스 연결 구조 정의
4. 통합 대시보드 구축
5. 옥동 시나리오의 생산성 향상 효과 강화
6. 이후 동적 워크플로우, 가상화, 스케줄링, 고도화 오프로딩으로 확장

즉, 지금 단계는 다음이 아니다.

* 전체 지능형 오케스트레이션 플랫폼을 먼저 완성하는 단계

지금 단계는 다음이다.

* 플랫폼이 실제로 붙고, 연결되고, 보이고, 효과가 있음을 먼저 입증하는 단계

## 3. 지금 플랫폼이 반드시 보여줘야 하는 것

플랫폼은 하나의 통합 흐름으로 아래를 보여줄 수 있어야 한다.

* 어떤 디바이스가 연결되어 있는지
* 어떤 엣지 노드가 활성 상태인지
* 어떤 서비스가 배치되어 실행 중인지
* 디바이스와 서비스가 어떻게 연결되어 있는지
* 어느 노드가 어떤 서비스를 실행 중인지
* 장애, 과부하, 미연결, 성능저하 상태가 있는지
* 데모가 어떤 운영 효율 또는 생산성 향상 효과를 제공하는지

대시보드는 단순 리소스 모니터가 아니라, 아래의 관계를 함께 보여주는 운영 뷰여야 한다.

* 센서 디바이스
* Jetson / Raspberry Pi / 서버 노드
* 가상 디바이스 / 디바이스트윈
* AI 서비스
* 서비스 배치 및 실행 상태

## 4. 현재 최우선 구현 항목

### 4.1 서비스 데모 1종 우선 완성

대표 산업 시나리오 1종을 먼저 선택하고, 이를 실제로 구동 가능한 데모로 완성한다.

데모는 반드시 아래를 포함해야 한다.

* 디바이스 입력
* 서비스 실행
* 결과 전달
* 대시보드 표시
* 운영 상태 확인

데모는 단순 기능 시연이 아니라 아래 질문에 답해야 한다.

* 무엇이 기존보다 빨라졌는가
* 어떤 운영 부담이 줄었는가
* 왜 이 서비스가 현장에 필요한가

### 4.2 디바이스 등록 및 관리 체계 정리

현재 중요한 것은 매퍼 자체보다 디바이스 관리 체계다.

반드시 정리할 것:

* 디바이스 유형 분류

  * 센서 디바이스
  * 엣지 디바이스
  * 가상 디바이스
  * 디바이스트윈
* 신규 디바이스 등록 절차
* 등록 주체와 승인 절차
* 플랫폼에서 보일 메타데이터
* 디바이스 상태 모델

  * online / offline
  * telemetry 수신 여부
  * 서비스 연결 여부
  * 할당된 노드 / twin 정보

### 4.3 디바이스-서비스 연결 구조 구현

플랫폼 가치는 디바이스를 붙이는 것만으로는 충분하지 않다.
서비스와 실제로 연결되어야 한다.

반드시 구현할 것:

* 디바이스와 서비스의 바인딩 구조
* 입력 디바이스 → 처리 노드 → 서비스 → 결과 전달 흐름
* 서비스 실행 상태의 추적
* 디바이스 관점에서 어떤 서비스에 연결되었는지 조회 가능해야 함

### 4.4 통합 대시보드 구축

대시보드에서는 최소한 아래가 함께 보여야 한다.

* 엣지 서버 상태
* Jetson / Raspberry Pi 상태
* 센서 디바이스 상태
* 디바이스-서비스 연결 관계
* 서비스 배치 위치
* 장애 / 미연결 / 과부하 상태
* 시나리오별 처리 결과 및 KPI

권장 화면 구성:

1. 자산 현황 영역
2. 연결 관계 영역
3. 운영 상태 영역
4. 시나리오 KPI 영역

### 4.5 옥동 시나리오 보강

옥동 시나리오는 단순 탐지 데모 수준을 넘어서야 한다.

반드시 정리할 것:

* 어떤 공정의 어떤 문제를 개선하는가
* 기존 수작업 또는 단순 룰 기반 방식 대비 무엇이 나아지는가
* 운영자 개입이 얼마나 줄어드는가
* 어떤 생산성 지표로 설명할 것인가

생산성 KPI 예시:

* 응답 시간 단축
* 작업자 개입 횟수 감소
* 이상 대응 시간 감소
* 설비 가동 중단 감소
* 품질/안전 리스크 감소
* 현장 재설정 시간 감소

## 5. 기술 고도화 항목은 후순위

아래 항목들은 중요하지만 현재 1순위는 아니다.
데모와 연결 구조가 먼저다.

후순위 고도화 항목:

* 동적 워크플로우 분해 및 재배치
* 서비스 분할 및 오프로딩 고도화
* 컨테이너 기반 가상 디바이스 / 서비스 태스크 구조
* 런타임 리플래닝
* 엣지 단독 학습 상황 대응
* 자원 스케줄링 고도화
* CXL 등 확장 자원 활용
* agent-assisted planning layer

원칙:

* 고도화 기능은 데모 완성 이후 단계적으로 붙인다.
* 현재 단계에서 고도화 기능을 먼저 과장하지 않는다.

## 6. 현재 테스트베드 해석

현재 플랫폼의 실증 기반은 이미 일부 확보되어 있다.

현재 전제로 두는 노드:

* x86 서버 노드
* Jetson 엣지 디바이스
* Raspberry Pi 5 엣지 디바이스

현재 전제로 두는 구조:

* KubeEdge 기반 mixed-device 환경
* 디바이스트윈 개념 포함
* 오프로딩 / 서버 자원 추가 할당 개념 포함
* 대시보드 구성 개념 포함

즉, 현재 단계는 “완전한 설계만 존재하는 상태”가 아니라, 최소한의 테스트베드와 구조는 이미 있으며 이를 실제 데모 수준으로 끌어올려야 하는 단계다.

## 7. 구현 원칙

### 7.1 먼저 보이게 만든다

현재는 복잡한 제어 로직보다 “보이는 데모”가 더 중요하다.

### 7.2 디바이스 중심으로 본다

현재 플랫폼은 서비스만이 아니라 디바이스 등록, 연결, 상태, 서비스 관계를 수평적으로 보여줘야 한다.

### 7.3 기술 나열보다 운영 흐름을 우선한다

문서나 발표에서는 기술 이름을 늘어놓기보다 아래 흐름으로 설명해야 한다.

* 어떤 디바이스가 연결됨
* 어떤 서비스가 실행됨
* 어디서 처리됨
* 어떤 결과가 나옴
* 어떤 운영 효과가 생김

### 7.4 생산성 언어로 설명한다

“모델이 있다”보다 “현장 효율이 올라간다”가 중요하다.

### 7.5 동적 워크플로우는 장기 방향으로 유지한다

최종 목표는 동적 워크플로우 기반 플랫폼이다.
하지만 현재 단계에서는 그 전체를 완성했다고 포장하지 않는다.

## 11. 현 시점의 한 줄 정의

이 프로젝트의 현재 단계는,
**혼합 디바이스 엣지 AI 플랫폼의 복잡한 전체 기능을 먼저 완성하는 단계가 아니라, 디바이스와 서비스를 실제로 연결하고 이를 대시보드에서 가시화하며 현장 생산성 향상 효과를 설명 가능한 서비스 데모로 완성하는 단계**다.

장기적으로는 이를 기반으로 동적 워크플로우, 가상화, 스케줄링, 오프로딩 고도화로 확장한다.
