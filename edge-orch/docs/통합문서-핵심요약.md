# 통합문서 핵심요약

이 문서는 `통합문서.md`에서 실제 실행/연구에 필요한 핵심만 추린 요약본이다.

## 1) 프로젝트 한 줄 정의
- KubeEdge/Kubernetes 기반 **이기종 혼합 디바이스 환경**에서 AI 워크플로우를 단계 단위로 배치·마이그레이션·오프로딩하는 런타임 오케스트레이션.

## 2) 반드시 해결할 문제
- 노드 이기종성(`amd64/arm64`, GPU/메모리/네트워크 차이) 때문에 정적 배치가 빠르게 비효율화됨.
- 단일 서비스 단위 관리로는 병목 대응이 어렵기 때문에 **stage 단위** 제어가 필요함.
- 핵심 stage: `capture -> preprocess -> inference -> postprocess -> result delivery`

## 3) 연구/논문 중심축
- 중심 문장: **이기종 혼합 디바이스 엣지 AI 워크플로우를 위한 런타임 오케스트레이션**
- 핵심 축:
  1. workflow-stage decomposition
  2. runtime state collection/normalization
  3. stage-level placement/replanning/offloading
  4. real mixed-device testbed validation

## 4) 현재 구현 상태 (요약)
- 완료:
  - `state_aggregator` 기반 상태 수집/정규화
  - ArgoCD + Image Updater 기반 GitOps
  - multi-arch CI/CD
  - `workflow_executor` 이력 저장 및 실행 개선
  - heuristic 기반 `placement_engine`
- 다음 우선순위:
  1. 실제 AI 워크플로우 1개 강한 통합
  2. 단계 간 데이터 전달 경로 정리
  3. live 상태 기반 runtime replanning 강화

## 5) 디바이스 PoC 핵심 상태
- 검증 완료:
  - 가상 디바이스 컨테이너 병렬 실행 구조
  - `temp-device-01` 단일 디바이스의 `DeviceStatus` reported 반영
- 남은 병목:
  - 20개 병렬에서 `mqttvirtual` 매퍼 불안정(`broken pipe`, `connection refused` 등)
  - 병목은 KubeEdge 코어보다 mapper 구현 구조 가능성이 높음

## 6) 핵심 구성요소 역할
- `state_aggregator`: 메트릭/이벤트 수집, 상태 정규화, 경량 API 
- `workflow_reporter`: stage 실행 이벤트 전송
- `placement_engine`: 단계별 배치/오프로딩 판단
- `agent_assisted_planner`: 과부하·예외 시 선택적 개입(후순위)

## 7) 운영 원칙
- 초기 제어는 heuristic 중심으로 단순/명시적으로 유지
- RL/LLM-first 제어는 초기 범위에서 제외
- Prometheus/Grafana는 재사용하고, 중복 모니터링 시스템은 만들지 않음

## 8) 지금 당장 실행 체크리스트
1. 실 워크플로우 1개를 end-to-end로 연결
2. stage 간 payload/handoff 포맷 고정
3. overload/burst 시 replanning 트리거 조건 명문화
4. 20개 병렬 디바이스 안정화(매퍼 구조 개선 우선)
